# 01 — نیازمندی‌ها و معیارهای پذیرش

## دامنه MVP

### FR-01 — افزودن دارو

کاربر باید بتواند موارد زیر را ثبت کند:

- نام دارو
- شکل/واحد موجودی: قرص، کپسول، میلی‌لیتر، دوز، قطره، ویال یا سایر
- مقدار موجودی در زمان ثبت
- مقدار مصرف روزانه مطابق دستور پزشک
- فاصله هشدار قبل از اتمام، با مقدار پیش‌فرض ۵ روز
- توضیح اختیاری

**معیار پذیرش:** نام خالی، موجودی منفی یا مصرف روزانه صفر/منفی ذخیره نشود.

### FR-02 — محاسبه موجودی

برنامه باید بر اساس مقدار موجودی ثبت‌شده، مصرف روزانه و زمان ثبت، مقدار تقریبی باقی‌مانده را محاسبه کند.

**معیار پذیرش:** برای ۳۰ قرص با مصرف روزانه ۲ قرص، در لحظه ثبت ۱۵ روز باقی‌مانده نمایش داده شود.

### FR-03 — تاریخ تقریبی اتمام

برنامه باید تاریخ و زمان تقریبی اتمام موجودی را نمایش دهد.

**معیار پذیرش:** محاسبه از زمان ثبت موجودی انجام شود و با گذشت زمان کاهش یابد.

### FR-04 — وضعیت اولویت

هر دارو یکی از وضعیت‌های زیر را داشته باشد:

- ایمن
- نزدیک به اتمام
- بحرانی
- تمام‌شده

قواعد اولیه:

- `تمام‌شده`: موجودی برآوردی صفر
- `بحرانی`: کمتر یا مساوی ۳ روز
- `نزدیک به اتمام`: کمتر یا مساوی فاصله هشدار کاربر
- `ایمن`: سایر موارد

### FR-05 — مرتب‌سازی داشبورد

داروها باید براساس زودترین زمان اتمام مرتب شوند.

### FR-06 — به‌روزرسانی موجودی

در نسخه بعدی MVP کاربر می‌تواند خرید جدید را ثبت کند. این عمل باید یک رویداد تاریخچه‌ای بسازد و مبنای محاسبه را به زمان جدید منتقل کند.

### FR-07 — هشدار محلی

برنامه باید بدون سرور بتواند قبل از پایان موجودی اعلان محلی ایجاد کند.

### FR-08 — آرشیو و حذف

داروی قطع‌شده باید قابل آرشیو باشد. حذف دائمی نیازمند تأیید کاربر است.

## نیازمندی‌های غیرعملکردی

### NFR-01 — Offline-first

تمام قابلیت‌های اصلی بدون اینترنت کار کنند. اینترنت فقط برای بارگذاری تبلیغ لازم است.

### NFR-02 — حریم خصوصی

نام دارو، مقدار مصرف و موجودی نباید در eventهای تبلیغاتی یا analytics ثبت شوند.

### NFR-03 — عملکرد

- زمان شروع سرد روی دستگاه میان‌رده هدف: کمتر از ۲.۵ ثانیه
- نمایش داشبورد پس از بازشدن دیتابیس: کمتر از ۳۰۰ میلی‌ثانیه
- محاسبه هر دارو: کمتر از ۵ میلی‌ثانیه

### NFR-04 — دسترس‌پذیری

- رابط کاملاً RTL
- حداقل اندازه ناحیه لمس ۴۸dp
- پشتیبانی از بزرگ‌نمایی فونت تا ۲۰۰٪ بدون قطع اطلاعات اصلی
- عدم اتکای انحصاری به رنگ برای نشان‌دادن وضعیت

### NFR-05 — قابلیت نگهداری

- منطق محاسبه مستقل از Flutter UI باشد.
- هر تغییر در فرمول موجودی تست واحد داشته باشد.
- SDK تبلیغات فقط از طریق `AdService` فراخوانی شود.

### NFR-06 — پایداری

هدف نسخه عمومی:

- Crash-free sessions بیشتر از ۹۹.۵٪
- ANR کمتر از ۰.۳٪

## سناریوهای لبه

1. مصرف روزانه اعشاری مانند نصف قرص.
2. موجودی کمتر از یک روز مصرف.
3. زمان دستگاه به گذشته تغییر کند.
4. کاربر موجودی را بعد از چند روز ویرایش کند.
5. مصرف دارو موقتاً متوقف شود.
6. مقدار موجودی بسیار بزرگ یا غیرواقعی وارد شود.
7. نبود اینترنت و شکست بارگذاری تبلیغ.

## تعریف Done برای هر قابلیت

- معیار پذیرش نوشته شده باشد.
- منطق دامنه تست واحد داشته باشد.
- حالت خالی، خطا و loading طراحی شده باشد.
- متن فارسی و RTL بررسی شده باشد.
- eventهای analytics فاقد داده پزشکی باشند.
- رفتار تبلیغات با سیاست محصول تطبیق داده شده باشد.
